# План автоматизации и тестирования мобильного приложения «Мобильный хоспис»

## 1. Цель тестирования
Проведение комплексного тестирования (ручного и автоматизированного) мобильного приложения «Мобильный хоспис» для проверки стабильности работы основного функционала, корректности UI-элементов, навигации и обработки негативных сценариев.

## 2. Окружение (Environment)
* **Устройство:** Эмулятор Pixel 4
* **Операционная система:** Android 10.0 (API 29)
* **Среда разработки:** Android Studio Quail 1

## 3. Виды тестирования
* **Функциональное тестирование:** проверка основных пользовательских сценариев (авторизация, просмотр и фильтрация новостей, работа с заявками).
* **UI-тестирование (Интерфейсное):** проверка корректности отображения элементов, кнопок, текстовых полей и переходов между экранами.
* **Тестирование локализации:** проверка корректности отображения текстовых данных на русском языке.
* **Негативное тестирование:** проверка поведения системы при вводе некорректных данных (пустые поля, невалидный пароль).

## 4. Инструменты (Инструментарий)
* **Android Studio** — основная среда разработки и запуска проекта.
* **UI Automator Viewer / Layout Inspector** — для инспектирования UI-элементов и поиска их ID / селекторов.
* **Espresso / UI Automator** — фреймворки для написания UI-автотестов.
* **Allure Framework** — для генерации наглядных отчетов о прохождении автотестов.

## 5. Техники тест-дизайна
Для обеспечения максимального тестового покрытия при ограниченных ресурсах будут использованы:
* **Классы эквивалентности:** разбиение входных данных на группы (валидные и невалидные), где проверка одного значения эквивалентна проверке всей группы.
* **Анализ граничных значений:** тестирование критических точек ввода (минимально/максимально допустимые значения), так как ошибки чаще возникают на границах.

## 6. Перечень и описание рисков при автоматизации
* **Ограниченность эмулятора:** эмулятор не полностью воспроизводит специфические прерывания реального мобильного ОС.
* **Изменение API:** модификация формата данных на сервере может сделать текущие автотесты неактуальными.

## 7. Необходимые специалисты
* **QA Engineer (Automation):** разработка и поддержка фреймворка, написание тестов, работа с Allure.
* **Manual QA:** подготовка данных, ручная проверка сложного UI/UX, верификация багов.
* **Developer (Mobile):** консультации по архитектуре, предоставление доступа к элементам приложения.

## 8. Сортировка тест-кейсов по степени автоматизации
* **Автоматизируемые (автотесты):** Основные бизнес-процессы, требующие частой проверки (авторизация, создание новости, фильтрация).
* **Ручные:** Проверка визуальных элементов (верстка, анимации, переходы), а также сценарии, работа с которыми заблокирована известными дефектами UI.

## 9. Интервальная оценка с учётом рисков
* **Анализ и планирование:** 15%
* **Разработка инфраструктуры:** 25%
* **Реализация автотестов:** 35%
* **Стабилизация (рефакторинг):** 10%
* *Примечание:* заложено +15% времени на непредвиденные проблемы с окружением.

## 10. Тестовая документация
* **Ручная часть:** 
    * Чек-лист (общий перечень проверок).
    * Тест-кейсы (детальное описание шагов и ожидаемых результатов).
    * Баг-репорты (реестр выявленных дефектов, оформленный в файле `Cases.xlsx` на отдельном листе).
* **Автоматизированная часть:** 
    * `README.md` (инструкция по запуску автотестов).
    * Отчеты `Allure` (результаты выполнения с шагами).
    * `Result.md` (сравнительный анализ ручного и автоматизированного тестирования).
